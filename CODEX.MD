# CODEX

## Назначение файла
- `CODEX.MD` — единый источник правил, ограничений и рабочих соглашений для Codex в этом проекте.
- История изменений, прошлые решения и накопленный прогресс ведутся в `CHANGELOG.md`.
- Перед планированием, изменением кода и запуском команд нужно читать `CODEX.MD` и `CHANGELOG.md`.

## Базовые ограничения проекта
- Расширение остаётся MVP и не должно эволюционировать в production-grade сетевой инспектор.
- Сбор сетевых данных разрешён только после явного вооружения (`arm`) текущего `http/https` домена пользователем.
- Пользователь должен иметь явный UI для включения и выключения armed-состояния текущего домена без ручной правки storage.
- После вооружения домена сбор должен переживать перезагрузку страницы, чтобы захватывать ранние запросы этого домена.
- По умолчанию глобальный фоновый захват на всех сайтах запрещён.
- Данные не отправляются наружу: разрешены только локальные механизмы расширения (`postMessage`, `runtime.sendMessage`, `storage.local`, открытие extension tab).
- `latestCapture` считается одноразовым сообщением и должен удаляться после чтения или по TTL.
- Пользователь должен иметь локальную кнопку копирования `latestCapture` без обязательного открытия viewer.
- Дефолтный export во viewer должен оставаться компактным и пригодным для LLM: предпочтение у `DOM clean` и структурной схемы API, а не у полного raw payload.
- Если viewer или popup вынуждены работать с compact fallback вместо полного capture, это должно быть явно показано пользователю, а не скрыто молчаливым truncation.
- DOM preview во viewer должен иметь стабильный fallback на `cleanHtml`; sandboxed стилизованный preview допускается только как дополнительный best-effort слой.
- Если для capture доступен `cloneSpec`, viewer должен показывать пользователю найденные связи `DOM ↔ API` (`bindings`/`dataRequirements`) с `JSON path` и источником endpoint, а не ограничиваться только списком выбранных API.

## Проверка изменений
- После изменений обязательно проверять, что расширение не делает собственных внешних HTTP-запросов.
- После любых изменений кода обязательно запускать `npm test`; для чисто документальных правок этот прогон не обязателен.
- Для security-правок приоритет у совместимости страницы: capture не должен ломать `fetch`/`XHR`, даже если часть данных будет пропущена.
- При review обязательно отдельно оценивать: границы включения capture, подделку событий page→content и срок жизни данных в storage.

## Обязательная фиксация изменений
- Каждая новая фича обязана обновлять `README.md`, `CHANGELOG.md` и `CODEX.md`.
- После любых изменений в коде Codex обязан обновлять проектную документацию и фиксировать принятые решения.
- Минимум нужно обновлять `CHANGELOG.md`, если поведение, UI, ограничения или рабочие соглашения изменились.
- Этот шаг обязателен и не может пропускаться даже для небольших правок.

## Ограничения среды
- `CODEX-RULES.MD` в репозитории отсутствует; источником локальных правил считаются `CODEX.MD`, `AGENTS.md` и `CHANGELOG.md`.
